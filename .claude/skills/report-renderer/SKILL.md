# Skill: Report renderer

The markdown report is the primary human-readable output of Foxhole. It must read like it was written by a senior engineer, not generated by a linter. Tone is direct and professional. Language is plain.

## Section order

`renderMarkdownReport` assembles sections in this exact order:

1. Title and metadata
2. Summary paragraph
3. Score
4. Categories (single-page) or Results by page (multi-page)
5. Prioritized fixes
6. Findings
7. Performance metrics (only when `perf` check was run)
8. Run details

## Title and metadata

```markdown
# Foxhole Audit Report

**URL:** https://example.com
**Audited:** 2024-03-15T14:32:01.000Z
**Duration:** 18.4s
```

For multi-page runs, the URL line is replaced with a bulleted list of audited pages.

## Summary paragraph

The summary is a single paragraph in plain language. It states the overall condition of the frontend, names two or three specific issues, and ends with the single highest-priority action.

Rules:

- No bullet points.
- No WCAG clause references.
- No Lighthouse audit ID references.
- State the score. If a threshold was set, state whether it passed.
- Name specific problems, not categories. "Six product images are missing alt text" not "accessibility issues were found".

## Score

```markdown
## Score

**74 / 100** - Below threshold (80)
```

Pass/fail label is only shown when a threshold was set. Without a threshold:

```markdown
## Score

**86 / 100**
```

If the `perf` check ran, append the Lighthouse performance score on a second line:

```markdown
Lighthouse performance score: 78.
```

## Categories

```markdown
## Categories

| Category      | Score | Critical | Major | Minor |
| ------------- | ----- | -------- | ----- | ----- |
| Accessibility | 61    | 2        | 3     | 1     |
| Performance   | 78    | 0        | 2     | 4     |
| Semantic HTML | 90    | 0        | 0     | 2     |
| Bundle        | 85    | 0        | 1     | 1     |
```

Errored categories show `errored` in the score column with dashes for counts. Skipped categories are omitted from the table entirely.

For multi-page runs, replace with `## Results by page`, one `### {url} - score {n}` subsection per page, each with its own category table.

## Prioritized fixes

```markdown
## Prioritized fixes

### 1. Add alt text to product images

**Effort:** Low | **Severity:** Critical | **Category:** Accessibility

Six product images have no alternative text. Screen reader users receive no
information about these images.

Resolves: a11y1b2c3d4e5f6a7b, a11y8c9d0e1f2a3b4c

---
```

`Resolves` lists finding IDs, not rule IDs.

## Findings

Group by category. Within each category, sort by severity (critical first).

```markdown
## Findings

### Accessibility

#### [Critical] Images missing alternative text

Six product images have no alternative text.

**Selector:** `img.product-image`
**WCAG:** 1.1.1
**Recommendation:** Add descriptive alt text to each image.

---

#### [Critical] Images missing alternative text (framework)

Same heading with `(framework)` appended when `finding.kind === "framework"`.
```

Fields rendered conditionally:

- `**Selector:**` only when `finding.selector` is non-null
- `**WCAG:**` only when `finding.wcag` is non-null
- `**Source:**` `file:line:column` when `finding.source` is non-null

Backticks in selectors are escaped to prevent markdown injection.

## Performance metrics

Only rendered when `meta.checks_run` includes `"perf"`.

```markdown
## Performance metrics

| Metric            | Value    |
| ----------------- | -------- |
| LCP               | 4,200ms  |
| FID               | 320ms    |
| FCP               | 1,800ms  |
| TBT               | 340ms    |
| CLS               | 0.12     |
| TTFB              | 420ms    |
| Performance score | 78       |
| Bundle size       | 842.0 KB |
```

When `framework_bundle_size` is non-null, the bundle row splits into three:

```markdown
| Framework bundle size | 380.0 KB |
| Application bundle size | 462.0 KB |
| Total bundle size | 842.0 KB |
```

Null metric values are omitted entirely from the table.

## Run details

```markdown
## Run details

|                 |                              |
| --------------- | ---------------------------- |
| Foxhole version | 1.0.0                        |
| Node version    | 20.11.0                      |
| Platform        | darwin                       |
| Input mode      | url                          |
| Checks run      | perf, a11y, semantic, bundle |
| Duration        | 18,400ms                     |
```

## Tone and language rules

- Sentence case everywhere. No title case headings.
- Plain ASCII only. No em dashes, no smart quotes.
- No passive voice in recommendations. "Add alt text" not "Alt text should be added".
- No hedging. "This will improve FCP" not "This may potentially help with FCP".
- Numbers in findings are specific. "Six images" not "several images".
- Metric values include units. `4,200ms` not `4200`.
- Bundle sizes formatted as `{n} KB` with one decimal place.

## Anti-patterns

**failure**: Rendering findings before prioritized fixes, or metrics before findings. Section order is fixed by `renderMarkdownReport`.

**failure**: Showing Pass/Fail when no threshold is configured. The score label only appears when `report.meta.threshold !== null`.

**failure**: Including a "Bundle size" row when `framework_bundle_size` is non-null. Use the three-row split instead.

**warning**: Including `perf_runs`, `concurrency`, or `exclude_framework` in the Run details table. These fields exist in `RunMeta` but are not rendered by `renderMarkdownReport`.

**warning**: Rendering em dashes or smart quotes in any report content. Run output through the plain-ASCII rule even in `summarizeReport`.

**info**: Omitting the `(framework)` suffix on bundle findings with `kind === "framework"`. This is how users distinguish framework from application findings in multi-source bundle reports.

## When to load this skill

- Working in `src/report/markdown.ts`
- Working in `src/audit/summarize.ts` (the summary paragraph must follow the same tone rules)
- Reviewing or updating markdown report snapshot tests

## When not to load this skill

- Working in `src/runner/`, `src/audit/score.ts`, or `src/cli/`
- Working in `src/mcp/` (MCP tools return raw JSON, not rendered markdown)
