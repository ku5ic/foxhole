# ADR-008: No automatic crawling in v1, explicit URL list for SPA support

**Status:** Accepted
**Date:** 2026-04-07

## Context

Frontend audit tools often include a crawler that discovers pages automatically by following links from a seed URL. For SPA applications that render routes client-side, crawling is more complex: the crawler must execute JavaScript, wait for route transitions, and identify navigable routes from the client-side router rather than from `<a href>` links in the initial HTML.

Building a reliable SPA-aware crawler is a significant engineering effort. It also introduces new failure modes: infinite loops, duplicate page detection, handling authenticated routes, and managing crawl depth limits.

## Decision

v1 does not include a crawler. Users provide the list of URLs or paths to audit explicitly via `--url` (single) or `--urls` (comma-separated list). For `--build` mode, paths are relative to the local server root.

Crawling is deferred to v2.

## Consequences

Users of SPA applications must know their routes in advance and list them explicitly. This is a reasonable requirement for CI integration, where the set of routes is known and stable.

The explicit list approach has advantages: no accidental audit of routes the user did not intend to include, no crawl depth limits to configure, no risk of the crawler getting stuck in an infinite loop, and deterministic audit scope.

The `--urls` flag accepts a comma-separated list: `--urls /login,/dashboard,/settings`. Combined with `--build`, this covers the primary SPA CI use case without a crawler.

The v2 crawler will be additive. The `--urls` flag will remain the explicit override. Users who want deterministic scope will continue to use it even after crawling is available.
