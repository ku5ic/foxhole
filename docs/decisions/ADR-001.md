# ADR-001: Use axe-core and Lighthouse as complementary audit engines

**Status:** Accepted
**Date:** 2026-04-07

## Context

Foxhole needs to audit frontend pages for accessibility violations and performance regressions. Several tools exist in this space. The key question is whether to use one engine that covers both concerns or two specialized engines.

## Decision

Use axe-core for accessibility and Lighthouse for performance. Run both against every page when both checks are requested.

## Consequences

axe-core is the industry standard for automated accessibility testing. It runs against the live DOM in a real browser context, catches violations that static analysis cannot, and maps findings directly to WCAG success criteria. No other tool matches its coverage for DOM-level a11y checks.

Lighthouse is the industry standard for web performance auditing. It produces Core Web Vitals, prioritized opportunities, and a scored report. It is maintained by Google and trusted by the broader ecosystem.

The two tools are not redundant. axe-core does not measure performance. Lighthouse includes a basic accessibility score but does not produce the granular, WCAG-mapped violations that axe-core does. Running both gives Foxhole complete coverage neither tool provides alone.

The cost is two separate audit passes per page. Lighthouse currently requires its own Chrome instance separate from the Playwright browser Foxhole uses for axe-core and semantic checks. This is tracked as a known limitation in ADR-005 and the README.
