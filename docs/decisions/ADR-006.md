# ADR-006: MCP server uses stdio transport

**Status:** Accepted
**Date:** 2026-04-07

## Context

The MCP specification supports multiple transport mechanisms. The two practical options for a local developer tool are stdio (reading from stdin, writing to stdout) and HTTP with Server-Sent Events (SSE). The choice affects how Claude Code and other MCP clients connect to Foxhole.

## Decision

Use stdio transport. `foxhole mcp` starts a process that reads MCP messages from stdin and writes responses to stdout. Claude Code spawns the process directly using the config:

```json
{
  "mcpServers": {
    "foxhole": {
      "command": "foxhole",
      "args": ["mcp"]
    }
  }
}
```

## Consequences

Stdio transport requires no port management, no authentication, no network configuration, and no firewall exceptions. The MCP client (Claude Code) owns the process lifecycle. When Claude Code exits, the Foxhole MCP process exits with it.

This is the correct transport for a local CLI tool. HTTP/SSE transport is appropriate for hosted services where multiple clients connect to a shared server. Foxhole in v1 is a local tool with a single caller per process.

The tradeoff is that stdio transport cannot serve multiple clients simultaneously. If a use case emerges where multiple agents need to share a single Foxhole MCP server instance, HTTP transport would need to be added. This is not a v1 requirement.

One side effect: because stdout is the MCP transport channel, all progress output from the runners must go to stderr. This constraint is enforced in `src/runner/index.ts` and documented in `CLAUDE.md`.
