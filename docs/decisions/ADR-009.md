# ADR-009: Lighthouse finding mapping, scoreDisplayMode filtering, and the accessibility category boundary

**Status:** Accepted
**Date:** 2026-05-24

## Context

ADR-001 established axe-core and Lighthouse as complementary audit engines. ADR-002 defined how raw engine output normalizes into the canonical `Finding` type, including the catalog-gap severity fallback for Lighthouse audits (score below 0.5 maps to critical, 0.5 to 0.89 maps to major, 0.9 and above produces no finding).

The first real Lighthouse run against a live URL exposed three defects in the mapping path that the unit tests did not catch, because the tests asserted on `Finding` shape rather than on semantic correctness against real Lighthouse output.

First, the runner treated every Lighthouse audit that was not a clear pass as a finding. The filter dropped audits with a numeric score at or above 0.9 and kept everything else, including audits with a null score. In Lighthouse a null score does not mean failure. It means the audit had nothing to evaluate. Audits carry a `scoreDisplayMode` that distinguishes real scored results (`binary`, `numeric`) from results that are not pass/fail signals (`notApplicable`, `informative`, `manual`, `metricSavings`). On a clean page, dozens of accessibility and best-practices audits return `notApplicable` because the page has no `<video>`, no `<object>`, no ARIA `meter`, and so on. The runner reported all of them as critical findings. A single audit of a healthy page produced 68 findings, 63 of them critical, against a page that Chrome DevTools scored 99 performance, 100 accessibility, 100 best practices.

Second, the runner labelled every Lighthouse finding with `category: "perf"` regardless of which Lighthouse category the audit belonged to. Accessibility audits (`aria-*`, `frame-title`, `valid-lang`, `tabindex`, and others) and best-practices audits surfaced under the performance category. The Foxhole category boundary did not reflect the Lighthouse category boundary.

Third, and following from the first two, the catalog had no entries for most of the audit IDs the runner was emitting, so the majority of findings fell through to the generic "review this audit in the Lighthouse documentation" recommendation. Most of those audit IDs should never have been findings in the first place, so this defect is largely downstream of the first.

## Decision

### Filter audits by scoreDisplayMode before severity mapping

Audit-to-finding conversion gains a precondition that runs before the severity mapping defined in ADR-002. An audit becomes a candidate finding only when its `scoreDisplayMode` is `binary` or `numeric`. Audits with `scoreDisplayMode` of `notApplicable`, `informative`, `manual`, or `metricSavings` are never findings, regardless of score.

For candidate findings, the existing ADR-002 rule applies: a numeric or binary score at or above 0.9 is a pass and produces no finding; below 0.9 produces a finding with severity from the catalog or the ADR-002 fallback.

This refines ADR-002 rather than replacing it. ADR-002's severity table is correct; it simply assumed all inputs were real scored audits. The `scoreDisplayMode` gate makes that assumption explicit and enforced.

### Map Lighthouse findings to Foxhole categories by Lighthouse category

The runner walks `lhr.categories[*].auditRefs` to determine which Lighthouse category each audit belongs to, and maps that to a Foxhole `CheckCategory`. Lighthouse performance audits map to `perf`. The handling of Lighthouse accessibility audits is covered by the next decision.

### axe-core is authoritative for accessibility; Lighthouse accessibility findings are dropped

Lighthouse accessibility audits are not converted into findings. The accessibility category is populated exclusively by axe-core.

The rationale: Lighthouse's accessibility category is itself powered by axe-core internally. Running axe-core directly through Playwright yields strictly more information than letting Lighthouse re-run it, including richer node selectors, impact metadata, and a current axe-core version under our own control. Allowing both paths to populate the `a11y` category would produce duplicate findings for the same underlying issue and require a dedup strategy keyed on rule equivalence across two different ID schemes, which is complexity with no upside.

This narrows ADR-001. axe-core and Lighthouse remain complementary across the tool as a whole, but for the accessibility category specifically the engines do not overlap: axe-core owns it, Lighthouse defers.

Lighthouse best-practices audits are out of scope for v1 and are also dropped. v1 has no Foxhole category that corresponds to Lighthouse best practices, and inventing one is deferred until there is a concrete use case.

## Consequences

Finding counts drop sharply and reflect real failures. The category boundary in Foxhole output matches the engine that produced each finding, with accessibility coming solely from axe-core.

The catalog now needs entries only for the finite set of real Lighthouse performance audits that can fail, rather than the full audit surface. That backfill is bounded and is part of the same change.

We lose Lighthouse's own accessibility and best-practices signals. The accessibility loss is immaterial because axe-core covers the same ground more thoroughly. The best-practices loss is a deliberate v1 scope decision and can be revisited.

Phase 1 was marked complete on the basis of schema-valid output. This ADR documents that schema validity was necessary but not sufficient: semantic correctness against real runner output was never validated, and three defects survived into the codebase as a result. The Phase 2 validation criteria should include at least one integration assertion that runs Lighthouse against a known-good fixture page and checks that a clean page produces few or no findings, not merely that the findings are well-shaped.
