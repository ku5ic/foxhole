# ADR-002: Finding normalization schema and stable ID format

**Status:** Superseded by docs/spec/schemas.md sections 1.3, 2 and docs/spec/architecture.md section 5.3
**Date:** 2026-04-07

## Context

Foxhole wraps two audit engines that produce incompatible raw output formats. axe-core produces violation objects with impact strings and node selectors. Lighthouse produces audit objects with scores and opportunity details. Consumers of Foxhole output (both human report renderers and MCP tool callers) need a single, stable shape to work with.

Additionally, the diff feature requires that the same issue on the same page produces the same identifier across separate runs. Without stable IDs, regressions and improvements cannot be detected reliably.

## Decision

All runners normalize their raw output into the canonical `Finding` type before returning. No raw engine output leaves the runner layer.

Finding IDs are stable hashes computed per docs/spec/schemas.md section 2.

Field-level rules for `Finding` (including `rule_id` and `source`) are defined in docs/spec/schemas.md section 1.3.

Severity, effort, and recommendation come from the findings catalog (docs/spec/schemas.md section 8). When a vendor rule is not in the catalog, the runner falls back to a default severity (axe `critical`/`serious` -> `critical`, `moderate` -> `major`, `minor` -> `minor`; Lighthouse score below 0.5 -> `critical`, 0.5-0.89 -> `major`, 0.9+ -> no finding) and a generic recommendation, and emits a debug-level catalog-gap warning per architecture spec section 5.3.

## Consequences

Consumers of `Finding[]` do not need to know which engine produced a finding. The diff logic can match findings across runs by ID without engine-specific knowledge. The report renderer works against a single type regardless of which checks were run.

The tradeoff is that normalization loses some engine-specific detail. Raw axe-core output includes multiple affected nodes per violation; the normalized Finding captures only the first selector. This is acceptable for v1. If detailed per-node output is needed in future, it can be added as an optional field without breaking the existing schema.
