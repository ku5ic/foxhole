# ADR-007: Monetization boundary is the --push flag

**Status:** Accepted
**Date:** 2026-04-07

## Context

Foxhole needs a sustainable monetization model that does not gate useful functionality behind a paywall. Developer tools that hobble their free tier are resisted by the community and rarely achieve the distribution needed for a business to form around them.

The hosted layer requires infrastructure (database, storage, job queue) that has real costs. The question is where to draw the line between the free CLI and the paid hosted product.

## Decision

The CLI and MCP server are open source and free with no artificial limitations. All four checks, all output formats, unlimited local runs.

The monetization boundary is a single flag: `--push`. When a user runs `foxhole run --url https://example.com --push`, the local audit result is pushed to the Foxhole hosted service, which stores it and makes it available in the web dashboard, trend charts, and shareable report URLs.

The first `--push` from a new machine prompts for an API key. That is the entire conversion funnel.

Nothing behind the `--push` flag is implemented in the current codebase. The flag does not exist yet. It will be introduced when the hosted infrastructure is ready.

## Consequences

The free CLI has genuine value on its own. Users can adopt it, integrate it into CI, and build workflows around it without ever paying. This maximizes distribution and word-of-mouth.

The hosted layer adds persistence and history, which require infrastructure the CLI cannot provide. The value proposition is clear: local runs are free, historical tracking and monitoring are paid.

The risk is that the CLI is useful enough that many users never need the hosted layer. This is acceptable. A widely used open source tool with a small paid tier is a better outcome than a restricted tool with a larger paid tier that nobody uses.

No code should be written toward `--push` or the hosted layer without an explicit decision to begin that work. Any PR that introduces hosted layer logic without that decision is out of scope.
