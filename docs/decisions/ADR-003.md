# ADR-003: Error handling strategy

**Status:** Accepted, amended 2026-05-31
**Date:** 2026-04-07

## Context

Foxhole operates as both a CLI tool and an MCP server. Errors need to surface cleanly to humans via stderr and exit codes, and to MCP clients as structured error responses. The codebase needs a consistent approach that works for both consumers without duplicating error handling logic.

Two options were considered during Phase 0 planning:

- Option A: Result type everywhere in runners. Functions return `Promise<Result<T>>`. Call sites unwrap explicitly.
- Option B: Typed error classes, throw in runners, catch at orchestrator boundaries.

## Decision

Use typed error classes (Option B). Runners throw typed errors. The runner orchestrator in `src/runner/index.ts` catches per-page failures, isolates them, and continues. The CLI top-level handler catches all unhandled errors and maps them to exit codes. The MCP tool handlers catch and return structured error responses.

Error classes (`FoxholeError`, `ConfigError`, `RunnerError`, `NetworkError`, `ReportError`) live in `src/errors.ts` because they couple types with runtime constructors. The `Result<T>` type definition lives in `src/types/` per the architecture spec rule that all shared type definitions live there. `Result<T>` is used for config loading and file operations where the failure is expected and the caller needs to branch on success or failure. It is not used in the runner layer.

## Consequences

Call sites in the runner layer are simpler: no unwrap boilerplate, no `if (!result.ok)` at every boundary. Per-page isolation is handled once in the orchestrator rather than at every runner call site.

The tradeoff is that runners that throw rather than return errors are slightly less explicit about their failure modes. This is mitigated by the typed error classes: a `RunnerError` at the orchestrator tells the caller exactly which layer failed and includes the original cause.

The mixed `Result` / throw approach means callers need to know which pattern a given function uses. This is resolved by convention: functions in `src/config/` and `src/report/` that do file I/O return `Result`. Functions in `src/runner/` throw. This distinction is documented here and in the error-handling skill file.

The relocation of `Result<T>` from `src/errors.ts` to `src/types/` is a Phase 1 implementation task.

## Amendment (2026-05-31)

Two parts of this decision did not hold through Phase 1 implementation.

**`Result<T>` relocation withdrawn.** `Result<T>`, `ok()`, and `err()` remain in `src/errors.ts`. Two factors make the relocation impractical: the type constraint `E extends FoxholeError` couples `Result<T>` with the error classes defined in the same file, and `src/types/index.ts` is generated entirely from Zod schemas via `z.infer` -- it cannot host hand-written type definitions without breaking the single-source-of-truth guarantee that file provides. The planned Phase 1 relocation task is withdrawn. `Result<T>`, `ok()`, and `err()` are exported from `src/errors.ts` and remain available for future use.

**Two-convention approach not implemented.** The Consequences section documents a distinction where functions in `src/config/` and `src/report/` return `Result` while functions in `src/runner/` throw. This distinction does not exist in the shipped code. All file I/O in `src/config/load.ts` and `src/audit/read-report.ts` throws typed errors (`ConfigError`) on failure rather than returning `Result`. The codebase converged on throw-everywhere, with typed error classes providing the explicit failure-mode information the original design expected `Result` to carry. `Result`, `ok`, and `err` currently have no callers.
