# ADR-003: Error handling strategy

**Status:** Accepted
**Date:** 2026-04-07

## Context

Foxhole operates as both a CLI tool and an MCP server. Errors need to surface cleanly to humans via stderr and exit codes, and to MCP clients as structured error responses. The codebase needs a consistent approach that works for both consumers without duplicating error handling logic.

Two options were considered during Phase 0 planning:

- Option A: Result type everywhere in runners. Functions return `Promise<Result<T>>`. Call sites unwrap explicitly.
- Option B: Typed error classes, throw in runners, catch at orchestrator boundaries.

## Decision

Use typed error classes (Option B). Runners throw typed errors. The runner orchestrator in `src/runner/index.ts` catches per-page failures, isolates them, and continues. The CLI top-level handler catches all unhandled errors and maps them to exit codes. The MCP tool handlers catch and return structured error responses.

The typed error hierarchy is defined in `src/errors.ts`: `FoxholeError` (base), `ConfigError`, `RunnerError`, `NetworkError`, `ReportError`. All errors include a `code` string and an optional `cause` for wrapping lower-level errors.

The `Result` type is defined in `src/errors.ts` and used for config loading and file operations where the failure is expected and the caller needs to branch on success or failure. It is not used in the runner layer.

## Consequences

Call sites in the runner layer are simpler -- no unwrap boilerplate, no `if (!result.ok)` at every boundary. Per-page isolation is handled once in the orchestrator rather than at every runner call site.

The tradeoff is that runners that throw rather than return errors are slightly less explicit about their failure modes. This is mitigated by the typed error classes: a `RunnerError` at the orchestrator tells the caller exactly which layer failed and includes the original cause.

The mixed `Result` / throw approach means callers need to know which pattern a given function uses. This is resolved by convention: functions in `src/config/` and `src/report/` that do file I/O return `Result`. Functions in `src/runner/` throw. This distinction is documented here and in the error-handling skill file.
