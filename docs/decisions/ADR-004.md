# ADR-004: Playwright wait strategy before running checks

**Status:** Accepted
**Date:** 2026-04-07

## Context

Foxhole audits single-page applications and server-rendered pages using a real Chromium browser. SPA frameworks render asynchronously after the initial HTML load. If an audit runner executes against the DOM before the application has finished rendering, it will produce incomplete or incorrect findings: missing elements, unresolved routes, partial content.

The question is how long to wait and what condition to wait for before handing the page to axe-core, Lighthouse, or the semantic checker.

## Decision

Wait for Playwright's `networkidle` event, then add a fixed 500ms buffer before any runner executes against the page.

`networkidle` fires when there have been no more than two active network connections for at least 500ms. This covers most SPA hydration patterns, lazy-loaded components that fetch data on mount, and font loading. The additional 500ms buffer accounts for synchronous rendering work that happens after the last network request completes.

This logic is implemented once in `src/runner/browser.ts` in the `waitForPageReady` function and called by the runner orchestrator before dispatching to individual runners.

## Consequences

Pages that are genuinely slow to render will add up to roughly one second of wait time per page (networkidle timeout plus buffer). For a list of 10 SPA routes, this adds around 10 seconds to the total audit duration. This is acceptable for an audit tool.

The fixed buffer is a heuristic. It will be insufficient for applications that perform heavy synchronous computation after the last network request. If a user reports missing findings on a known-slow application, the buffer can be made configurable via `foxhole.config.json` in a future iteration.

The 500ms buffer value is not arbitrary; it matches the networkidle internal threshold, making the total effective wait "two networkidle cycles" in the worst case.
