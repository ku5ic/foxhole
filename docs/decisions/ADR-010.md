# ADR-010: Scoring model -- severity-weighted exponential decay

**Status:** Accepted
**Date:** 2026-05-30

## Context

Foxhole v1 used a linear penalty model: each finding subtracted a fixed number of points from a starting score of 100 (critical -15, major -8, minor -2), floored at 0. The page score was then the mean of category scores for categories with `status: "ok"`.

Two defects emerged from this model:

**Saturation.** The linear penalty floored at 0. A page with seven or more criticals scored the same as a page with twenty. Adding findings stopped mattering past the floor. The score no longer reflected the actual finding count once the load was high enough.

**Empty-category inflation.** The page score was the mean of ok-category scores. A page with one critical accessibility finding and three perfectly clean categories (no findings) scored around 96 because the three clean categories diluted the damage. A page with one critical issue is not a 96-quality page. The averaging step hid the finding behind noise.

## Decision

Replace the flat penalty-floor model with a severity-weighted exponential decay curve. Change the page score to be computed directly from all findings on the page rather than the mean of category scores.

### Severity weights

| Severity | Weight |
| -------- | ------ |
| critical | 10     |
| major    | 4      |
| minor    | 1      |

Ratios: critical is 2.5x major and 10x minor. A single critical outweighs two majors plus two minors.

### Scoring formula

```
W    = sum of weights for all findings in the set
score = round(100 * exp(-W / SCORE_SCALE))
```

`SCORE_SCALE = 40` (named constant in `src/audit/score.ts`).

### Curve table

| Findings             | W   | Score |
| -------------------- | --- | ----- |
| 0                    | 0   | 100   |
| 1 minor              | 1   | 98    |
| 1 major              | 4   | 90    |
| 1 critical           | 10  | 78    |
| 1 critical + 1 minor | 11  | 76    |
| 2 criticals          | 20  | 61    |
| 5 criticals          | 50  | 29    |
| 10 criticals         | 100 | 8     |
| 20 criticals         | 200 | 1     |

### Page score vs. category scores

The page score is computed from **all findings on the page** in one curve call. Category scores (for the report table) use the same `scoreFromFindings` function over only that category's findings. One function, two call sites.

The report score is the mean of ok-page scores (unchanged from v1).

### Properties

- **Strictly monotonic:** every additional finding always lowers the score. No plateau.
- **Asymptotes to 0:** never truly reaches 0 until W -> infinity; no floor clamp needed.
- **Empty-category fix:** a page with one critical scores 78 regardless of how many categories ran clean.
- **Saturation fix:** each additional finding continues to move the score.

## Alternatives considered

**Pure log penalty.** `score = round(100 - k * log(W + 1))`. Achieves monotonicity but the shape is concave from the start -- the first finding has a large impact and subsequent ones have diminishing impact, which is the wrong signal for an audit tool where each additional finding is independently important.

**Keep category mean, reweight.** Reweight category scores by findings count. Rejected: the mean is the source of the inflation problem. Computing the page score from all findings via one curve is simpler and fixes inflation and saturation together.

**Widen to 128-bit finding IDs.** Not related to scoring; tracked separately.

## Consequences

- Every score in every existing report changes. This is pre-release; there are no external consumers.
- The weights and `SCORE_SCALE` are named constants in `src/audit/score.ts`. Retuning the curve requires changing two lines and recomputing fixtures.
- Tests assert strict monotonicity and the no-plateau property explicitly.
