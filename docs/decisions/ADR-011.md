# ADR-011: Framework bundle classification -- measure execution cost, surface actionability separately

**Status:** Accepted
**Date:** 2026-05-31

## Context

The bundle runner measures JavaScript payload by intercepting every network response the browser makes during page load, summing decoded sizes, and comparing the total against a 500 KB threshold. This accurately reflects what the browser downloads and parses.

Modern framework-based SPAs (Next.js, Nuxt, SvelteKit, Astro) ship two categories of JavaScript:

**Framework chunks.** React, the framework runtime, the router, server-component infrastructure. A minimal Next.js app without a single line of application code delivers roughly 130 KB of framework JS (react + react-dom, minified). A typical production app runs 200-400 KB of framework before any product code is added.

**Application chunks.** Page components, utilities, third-party libraries chosen by the product team. These are directly under the developer's control.

Both categories are executed by the browser on every page load. Neither is hypothetical cost.

The original implementation treats them identically. This produces two problems in practice:

**Unfair threshold for framework-based projects.** The 500 KB threshold was chosen as a general JS payload budget. A Next.js project that ships 420 KB of framework JS and 90 KB of app code is lean by any reasonable standard, yet it fires the `bundle/total-js-size` major finding on every page.

**Unactionable recommendation.** The finding recommendation reads "Split bundles, remove unused code, and lazy-load non-critical JavaScript." This is sound advice for application code and useless for `react-dom.production.min.js`. A developer auditing their site for the first time gets a major finding with advice that does not apply to the actual cause.

### What this is not

This question is different from the `<script nomodule>` exclusion (ADR-010 follow-up). Polyfill chunks delivered via `nomodule` are never executed by modern browsers; excluding them from measurement corrects the measurement itself. Framework chunks are fully executed -- excluding them would undercount real browser work. The issue here is classification and communication, not measurement accuracy.

## Decision

Framework chunks are counted toward the JS size total. The execution cost they represent is real and belongs in the score. What changes is how the bundle runner classifies each JS chunk and what the finding's `kind`, `description`, and `recommendation` fields say.

### Classification approach: URL fast-path, then content-signature fallback

Classification runs in two stages, composing at resource collection time:

**Stage 1 -- URL pattern matching (`classifyResource`).**
Each JS chunk URL path is matched against `FRAMEWORK_URL_PATTERNS`. If a pattern matches, the chunk is classified `"framework"` with zero body-read cost. This covers all frameworks that embed a recognizable prefix in chunk names (webpack production builds, Vite dev server, SvelteKit, Nuxt, Remix, Gatsby) as well as development-mode tooling artifacts.

**Stage 2 -- Content-signature scanning (`classifyByContent`).**
For chunks the URL path leaves unclassified (returning `"application"`), the response body is read and a bounded prefix slice (64 KB) is scanned for minification-surviving framework runtime markers. A match upgrades the chunk to `"framework"`. This covers Turbopack production builds and other configurations that emit fully-hashed chunk names (e.g. `0z_s30ati0q4r.js`) with no framework-specific prefix.

Classification is computed once in the response handler and stored in `ResourceInfo.kind`. `buildBundleFindings` reads `resource.kind ?? classifyResource(resource.url)`, so it remains a pure function and existing tests that pass `ResourceInfo` without `kind` continue to exercise the URL-only path.

### URL patterns (`FRAMEWORK_URL_PATTERNS`)

Substring matches against the URL path, case-sensitive. Listed in `src/runner/bundle.ts` as a named constant.

| Framework               | Pattern                           |
| ----------------------- | --------------------------------- |
| Next.js                 | `/_next/static/chunks/framework`  |
| Next.js                 | `/_next/static/chunks/main-`      |
| Next.js                 | `/_next/static/chunks/pages/_app` |
| Next.js                 | `/_next/static/chunks/webpack`    |
| Create React App        | `/static/js/runtime-main`         |
| Vite (pre-bundled deps) | `/node_modules/.vite/`            |
| Vite (vendor split)     | `/assets/vendor-`                 |
| Vite dev client         | `/@vite/`                         |
| Vite React Fast Refresh | `/@react-refresh`                 |
| Nuxt 3                  | `/_nuxt/entry.`                   |
| Nuxt 3                  | `/_nuxt/builds/meta.`             |
| SvelteKit               | `/_app/immutable/entry/`          |
| SvelteKit               | `/_app/immutable/start-`          |
| SvelteKit dev           | `/@sveltejs/kit/`                 |
| Gatsby / webpack        | `/webpack-runtime-`               |
| Remix                   | `/build/entry.client-`            |

**`react-` is intentionally absent.** A `/_next/static/chunks/react-*.js` chunk is more likely an application code-split named after a React component than the React runtime itself; the React runtime ships inside `framework-*.js`. A unit test locks this to prevent unintentional reversion.

### Content signatures (`FRAMEWORK_CONTENT_SIGNATURES`)

String literals that survive minification because they are error-link prefixes, property keys, or internal sentinels preserved by minifiers even when variable names are mangled. A match on the first 64 KB of the chunk body is sufficient.

| Signature                                              | Framework | Notes                                                              |
| ------------------------------------------------------ | --------- | ------------------------------------------------------------------ |
| `"Minified React error #"`                             | React     | Error URL prefix emitted many times in react-dom production builds |
| `"__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED"` | React     | Internal sentinel, preserved as a property key                     |
| `"__vue_app__"`                                        | Vue 3     | Set only inside `createApp` mount logic, not in component code     |
| `"__sveltekit_"`                                       | SvelteKit | Runtime global prefix, not present in Svelte component code        |

**Angular is not covered by content signatures.** Angular's Ivy function names (e.g. `ɵɵdefineComponent`) appear in compiled app components as well as in the runtime, making them unreliable for chunk-level classification. Angular apps benefit from URL pattern coverage where chunk names are semantic; hashed Angular production builds currently receive the honest-gap note (see below).

### Framework naming (`detectFramework` and `FRAMEWORK_ROOT_PATTERNS`)

A separate helper identifies which framework is running by matching the JS URL list against distinctive static-root path prefixes. This operates independently of per-chunk classification and is used only to name the framework in the `bundle/total-js-size` description when `frameworkBytes === 0`.

Patterns are checked in priority order across all URLs (pattern-first, not URL-order-first), so framework-specific roots take precedence over the generic Vite bundler root even when Vite dev URLs load first.

| Pattern            | Name      |
| ------------------ | --------- |
| `/_next/`          | Next.js   |
| `/_nuxt/`          | Nuxt      |
| `/_app/immutable/` | SvelteKit |
| `/_astro/`         | Astro     |
| `/@vite/`          | Vite      |

### Three-way `bundle/total-js-size` description

1. `frameworkBytes > 0`: `"Total JavaScript transferred is X KB (Y KB framework, Z KB application), which exceeds the 500 KB threshold."` -- byte breakdown from URL or content classification.
2. `frameworkBytes === 0` and a framework root is detected: `"Total JavaScript transferred is X KB, which exceeds the 500 KB threshold. Detected a <Name> app; a framework and application breakdown is not available for this build's chunk naming."` -- honest-gap note for builds whose chunks could not be classified even by content (e.g. Angular).
3. Otherwise: `"Total JavaScript transferred is X KB, which exceeds the 500 KB threshold."` -- non-framework app or no detection.

### `bundle/large-javascript-chunk` finding

Sets `kind: "framework"` when the oversized chunk classifies as framework (by URL or content), and `kind: "application"` otherwise. The `recommendation` is replaced with framework-specific guidance for framework chunks: `"This is a framework chunk. Consider whether the chosen framework fits the project's size budget, or enable more aggressive tree-shaking via the bundler config."`

### Score unchanged

The score impact of framework findings is unchanged. A framework chunk that exceeds 200 KB still produces a minor finding with the same weight in the scoring formula. The score reflects actual payload cost.

### Threshold note

The `threshold` config field gives project teams control over what score they will accept. Teams on heavy framework stacks that accept the framework cost can raise their threshold in `foxhole.config.json`. No default threshold change is made here.

## Alternatives considered

**Exclude framework chunks from measurement.** Rejected. Excluding executed bytes from a performance audit is misleading. A page that delivers 600 KB of framework JS is slower to parse and execute than one that delivers 300 KB, regardless of who wrote the code. The score should reflect this.

**Separate framework and application thresholds.** Two thresholds would require new config schema fields and more complex finding logic. The complexity cost is not justified; the description-level breakdown achieves the same communication goal.

**Suppress framework findings below a configurable allowance.** An `allowedFrameworkKb` config field that subtracts from the total before the threshold check. Rejected: invisible budget adjustments make scores non-comparable across projects.

**Per-framework runtime manifest reading (`__BUILD_MANIFEST`, `__NUXT__`, etc.)** Most accurate where implemented, but one brittle, version-coupled reader per framework; scales poorly across many frameworks and breaks across major versions. Rejected.

**Content-signature scanning only (no URL fast-path).** Pays body-read and scan cost for every JS chunk including those whose names already classify exactly. Rejected; the URL fast-path is free for the common case.

## Consequences

- `src/runner/bundle.ts` gains `FRAMEWORK_URL_PATTERNS`, `FRAMEWORK_ROOT_PATTERNS`, `FRAMEWORK_CONTENT_SIGNATURES`, and `FRAMEWORK_CHUNK_RECOMMENDATION` constants; `classifyResource`, `classifyByContent`, `detectFramework`, and `measureJsWithClassification` helpers; and an optional `kind` field on `ResourceInfo`.
- `buildBundleFindings` is a pure function that consumes pre-classified `ResourceInfo`; it reads `resource.kind ?? classifyResource(url)` so it is backward-compatible with callers that omit `kind`.
- `Finding.kind` is in the schema as `"framework" | "application" | null`. This was a breaking change: all finding constructors across all four runners emit the field, and old JSON reports fail `read-report` validation. Accepted as pre-release cost (no external consumers).
- The framework recommendation override is applied in the runner. The catalog structure is not changed.
- Content classification fetches the response body for URL-unmatched JS chunks that would otherwise use the Content-Length shortcut. The body-read cost is bounded by the existing 10 MB cap and the 64 KB scan slice.
