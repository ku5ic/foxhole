# ADR-011: Framework bundle classification -- measure execution cost, surface actionability separately

**Status:** Accepted
**Date:** 2026-05-31

## Context

The bundle runner measures JavaScript payload by intercepting every network response the browser makes during page load, summing decoded sizes, and comparing the total against a 500 KB threshold. This accurately reflects what the browser downloads and parses.

Modern framework-based SPAs (Next.js, Nuxt, SvelteKit) ship two categories of JavaScript:

**Framework chunks.** React, the framework runtime, the router, server-component infrastructure. A minimal Next.js app without a single line of application code delivers roughly 130 KB of framework JS (react + react-dom, minified). A typical production app runs 200-400 KB of framework before any product code is added.

**Application chunks.** Page components, utilities, third-party libraries chosen by the product team. These are directly under the developer's control.

Both categories are executed by the browser on every page load. Neither is hypothetical cost.

The current implementation treats them identically. This produces two problems in practice:

**Unfair threshold for framework-based projects.** The 500 KB threshold was chosen as a general JS payload budget. A Next.js project that ships 420 KB of framework JS and 90 KB of app code is lean by any reasonable standard, yet it fires the `bundle/total-js-size` major finding on every page.

**Unactionable recommendation.** The finding recommendation reads "Split bundles, remove unused code, and lazy-load non-critical JavaScript." This is sound advice for application code and useless for `react-dom.production.min.js`. A developer auditing their site for the first time gets a major finding with advice that does not apply to the actual cause.

### What this is not

This question is different from the `<script nomodule>` exclusion (ADR-010 follow-up). Polyfill chunks delivered via `nomodule` are never executed by modern browsers; excluding them from measurement corrects the measurement itself. Framework chunks are fully executed -- excluding them would undercount real browser work. The issue here is classification and communication, not measurement accuracy.

## Decision

Framework chunks are counted toward the JS size total. The execution cost they represent is real and belongs in the score. What changes is how the bundle runner classifies findings and what the finding's `source` and `recommendation` fields say.

### Framework URL pattern matching

After resource collection, the bundle runner classifies each JS resource as `"framework"` or `"application"` by matching its URL against a list of known framework path patterns. The initial pattern list:

| Framework        | Pattern                           |
| ---------------- | --------------------------------- |
| Next.js          | `/_next/static/chunks/framework-` |
| Next.js          | `/_next/static/chunks/main-`      |
| Next.js          | `/_next/static/chunks/pages/_app` |
| Next.js          | `/_next/static/chunks/webpack-`   |
| Create React App | `/static/js/runtime-main`         |
| Vite / generic   | `/node_modules/.vite/`            |

A resource URL matching any pattern is classified as framework. All others are application. Pattern matching is prefix or substring on the URL path, case-sensitive.

The pattern list lives as a named constant in `src/runner/bundle.ts` so it can be extended without touching logic.

### Finding changes

The `bundle/total-js-size` finding gains a breakdown in its `description` field when framework JS is present: `"Total JavaScript transferred is X KB (Y KB framework, Z KB application)."` The `recommendation` field remains unchanged -- the advice to split bundles and lazy-load is valid for the application portion.

The `bundle/large-javascript-chunk` finding sets `source: "framework"` when the oversized chunk is a framework resource. The `recommendation` is replaced with framework-specific guidance: `"This is a framework chunk. Consider whether the chosen framework fits the project's size budget, or enable more aggressive tree-shaking via the bundler config."`

The score impact of framework findings is unchanged. A framework chunk that exceeds 200 KB still produces a minor finding with the same weight in the scoring formula. The score reflects actual payload cost.

### Threshold note

The `concurrency` and `threshold` config fields already give project teams control over what score they will accept. Teams on heavy framework stacks that accept the framework cost can raise their threshold in `foxhole.config.json`. No default threshold change is made in this ADR; the correct default remains 0 (no enforcement unless configured).

## Alternatives considered

**Exclude framework chunks from measurement.** Rejected. Excluding executed bytes from a performance audit is misleading. A page that delivers 600 KB of framework JS is slower to parse and execute than one that delivers 300 KB, regardless of who wrote the code. The score should reflect this.

**Separate framework and application thresholds.** Two thresholds -- one for framework JS, one for application JS -- would require new config schema fields and more complex finding logic. The complexity cost is not justified by the improvement over the description-level breakdown. The single threshold with breakdown description achieves the same communication goal.

**Suppress framework findings below a configurable allowance.** An `allowedFrameworkKb` config field that subtracts from the total before the threshold check. Rejected: it introduces invisible budget adjustments that make scores non-comparable across projects and opaque to readers of a report.

**Detect framework via `window.__NEXT_DATA__` or framework-specific globals.** More reliable for framework detection than URL patterns, but runs at page evaluation time rather than at response collection time. URL patterns are available during resource collection and do not require an extra page.evaluate call.

## Consequences

- `src/runner/bundle.ts` gains a `FRAMEWORK_URL_PATTERNS` constant and a `classifyResource` helper.
- `buildBundleFindings` receives an additional parameter for the classified resource map, or classification is done inline before the call. The public function signature changes -- this is a breaking change to the bundle runner's exported API surface within the `src/runner/` boundary. Callers outside that boundary (none currently) would need to update.
- `Finding.source` is populated for framework chunks for the first time. The `source` field exists in the schema (`docs/spec/schemas.md`) and is typed as `string | null`; no schema change is required.
- Existing tests for `buildBundleFindings` pass unchanged resources; new tests cover the framework classification path.
- The `bundle/large-javascript-chunk` catalog entry gains a `framework_recommendation` field, or the override is applied in the runner. Decision deferred to implementation; the catalog structure is not changed by this ADR.
