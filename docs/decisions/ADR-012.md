# ADR-012: compare --threshold and documentation-as-built decisions for report, init, and quiet

**Status:** Accepted
**Date:** 2026-06-01

## Context

A doc-drift audit of the whole project (see scratch/doc-drift-foxhole-whole-project-20260601-2138.md) identified four places where docs/spec/v1.md described a CLI surface that the code did not implement:

1. `foxhole compare` -- the spec documented `--output`, `--out`, and `--threshold` flags. The command registered no options and always wrote JSON to stdout.
2. `foxhole report` -- the spec documented an `--out` flag for file output. The command implemented `--output <format>` (json or markdown) and wrote only to stdout.
3. `foxhole init` -- the spec documented a `--force` flag to overwrite an existing config without prompting. The command always prompted interactively via a y/N readline question.
4. Config file -- the spec's `foxhole.config.json` example included a `quiet` field. The config Zod schema has no `quiet` field and uses `.strict()`, so a config containing `quiet` throws a ConfigError.

Section 18 of the product spec requires an ADR for changes to section 5 (CLI surface). This ADR records both the one feature addition and the three documentation-as-built decisions.

## Decision

### compare --threshold (feature addition)

Implement `--threshold <n>` on `foxhole compare`. After diffing the two reports and writing the JSON diff to stdout, the command exits with code 1 when `diff.score_delta < threshold`. `score_delta` is `after.score - before.score` (schemas.md:227), already computed by `diffReports` and present on `RunDiff`.

Rationale: this is the one capability with direct CI value. Teams that store audit JSON as artifacts need a way to gate on score regressions across builds. The other two compare flags (`--output`, `--out`) add format and file-routing flexibility; that flexibility is a shell redirect away (`foxhole compare a.json b.json > diff.json`) and is not worth the build effort at this stage.

The implementation follows the `run.ts` exit-code pattern: a `handleCompare` function returns a `Promise<number>` (0 or 1); the action wrapper calls `process.exit(code)` on success and `process.exit(2)` on any `FoxholeError`. NaN threshold throws `ConfigError` consistent with run.ts threshold validation. `handleCompare` is exported for direct testing; the action is the single exit site.

### report --out (document as-built)

`foxhole report` supports `--output <format>` (json or markdown, default markdown) and always writes to stdout. Adding `--out` would involve file I/O in the command, which is already covered by shell redirection for CLI users and is not needed for the MCP path. Document the actual flag in the spec; note that file output is achieved via shell redirection.

### init --force (document as-built)

`foxhole init` prompts interactively when the config file exists. The non-interactive `--force` skip is a small feature but its main value is in scripted environments where the config file should never be silently overwritten. The interactive prompt is safe and predictable; the feature is not blocking any known workflow. Document the interactive behavior in the spec.

### config quiet field (document as-built)

`--quiet` suppresses progress output to stderr. It is a per-run flag that makes sense only at invocation time, not as a persistent project setting. Adding `quiet` to the config schema and threading it through `resolve-options.ts` is small but changes the resolution-order contract for a flag that is semantically per-invocation. Document that `--quiet` is CLI-only; correct the config shape example to omit it.

## Alternatives considered

**Build all four features to match the spec.** Rejected. Three of the four (report `--out`, init `--force`, config `quiet`) have low operational value relative to their build + test surface. Shell redirection handles report file output; interactive prompts are safe for init; per-invocation flags belong at the call site, not the config.

**Document everything, build nothing.** Rejected. `compare --threshold` has genuine CI value -- teams cannot gate on score regressions without it -- and the spec explicitly commits to this capability.

## Consequences

- `src/cli/commands/compare.ts` gains `--threshold`, a `handleCompare` export, and an explicit `process.exit(code)` in the action.
- `docs/spec/v1.md` sections 5.3, 5.4, 5.5, and 6.1 are corrected to match the implementation.
- `docs/spec/schemas.md` section 1.8 is updated to document the four optional RunMeta fields (`run_id`, `project_id`, `commit_sha`, `branch`) that exist in the Zod schema but were absent from the spec.
