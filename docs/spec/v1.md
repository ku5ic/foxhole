# Foxhole v1 product spec

Version: 1.0.0
Status: Draft, pending review
Owner: Sinisa
Last updated: 2026-04-30

This document is the single source of truth for what Foxhole v1 is and is not. Architecture, schemas, and per-phase implementation specs depend on this document. Changes here cascade.

## 1. Purpose

Foxhole is a frontend audit tool that runs accessibility, performance, semantic HTML, and bundle checks against URLs, local builds, or SPA route lists. It produces structured JSON that is consumed by both human-readable markdown reports and AI agents via an MCP server.

The free CLI is the product. Distribution and trust are the goal. Monetization is deferred to a hosted layer behind a single `--push` flag, which is out of scope for v1.

## 2. Goals

1. Produce audit output that reads like a senior engineer wrote it. Deterministic, scannable, actionable, no AI tells.
2. Point findings at exact source files and lines wherever source maps allow.
3. Run fast enough to belong in a developer's inner loop and in CI.
4. Expose the same audit capability through a CLI and an MCP server, sharing one orchestrator.
5. Ship as a single npm package installable globally or per project.
6. Stay open source and free with no artificial limitations.

## 3. Non-goals for v1

These are deliberately excluded. Each is deferred to a later version with a stated reason.

1. **Crawling.** Users provide URLs explicitly via `--url` or `--urls`. Auto-discovery of routes is a v2 feature. Reason: scope, and SPA crawling is a hard problem that deserves its own design.
2. **Server-side audit compute.** All audits run on the user's machine. The hosted layer is Phase 2 and beyond. Reason: zero infrastructure cost until paying users justify it.
3. **LLM-generated narrative.** All report prose is rendered from deterministic templates and an authored findings catalog. LLM synthesis is a paid hosted feature, not part of the CLI. Reason: cost, latency, and reliability. The CLI must work offline and in CI.
4. **Plugin system.** Runners and report renderers are extensible internally via interfaces, but no public plugin API ships in v1. Reason: premature abstraction. Wait for real third-party demand.
5. **GUI or web dashboard.** CLI and MCP only. The hosted dashboard is a Pro tier feature, not v1.
6. **Auto-fix or codemods.** Foxhole reports findings, it does not modify user code. Reason: scope and trust. Auto-fix is a separate product surface.
7. **Visual regression.** No screenshot diffing. Reason: out of scope.
8. **i18n of report output.** English only in v1. Reason: scope.

## 4. Personas and primary workflows

### 4.1 Local developer running ad-hoc audits

A developer working on a feature wants quick feedback on their changes before opening a PR. They run:

```
foxhole run --url http://localhost:3000/checkout
```

They expect a human-readable markdown report in stdout, finished in under 15 seconds, pointing at the files they need to fix.

### 4.2 CI gate on pull requests

A team adds Foxhole to CI to prevent regressions. They run:

```
foxhole run --build ./dist --urls /,/products,/checkout --threshold 80 --output json --out ./audit.json
```

They expect a non-zero exit code on threshold failure, a JSON report uploaded as a CI artifact, and a run completing in under 90 seconds for a 3-route audit.

### 4.3 AI agent calling MCP tools

A developer using Claude Code asks the agent to audit their app. The agent invokes:

```
mcp tool: run_accessibility_audit
input: { url: "http://localhost:3000" }
```

It expects a structured JSON response in under 5 seconds, with findings the agent can reason about and present to the user.

## 5. CLI surface

### 5.1 Commands

The CLI exposes four commands. No subcommands beyond these in v1.

```
foxhole run        Run an audit
foxhole compare    Diff two audit reports
foxhole report     Re-render a JSON report as markdown
foxhole init       Generate a foxhole.config.json scaffold in the current directory
```

### 5.2 `foxhole run`

Runs an audit and produces a report. Exactly one of `--url`, `--urls`, or `--build` must be provided. If `--build` is combined with `--urls`, the URL list is treated as paths relative to the served build.

Flags:

| Flag             | Type    | Default                            | Description                                                                                                                          |
| ---------------- | ------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| `--url`          | string  | none                               | Single target URL                                                                                                                    |
| `--urls`         | string  | none                               | Comma-separated list of URLs or paths                                                                                                |
| `--build`        | string  | none                               | Path to a static build directory, served on an internal port                                                                         |
| `--checks`       | string  | all                                | Comma-separated subset of `perf,a11y,semantic,bundle`                                                                                |
| `--output`       | enum    | `markdown`                         | `json` or `markdown`                                                                                                                 |
| `--out`          | string  | stdout                             | File path for the report                                                                                                             |
| `--config`       | string  | `./foxhole.config.json` if present | Path to a config file                                                                                                                |
| `--threshold`    | number  | none                               | Fail with exit code 1 if overall score drops below N                                                                                 |
| `--concurrency`  | number  | `1`                                | Number of pages to audit in parallel. Above 1, perf scores become less reliable                                                      |
| `--perf-runs`    | number  | `1`                                | Number of Lighthouse runs per page, median is reported. Higher reduces variance                                                      |
| `--perf-profile` | enum    | `standard`                         | `fast`, `standard`, or `mobile`. Controls Lighthouse throttling                                                                      |
| `--wait-for`     | string  | none                               | CSS selector to wait for before running checks, in addition to networkidle                                                           |
| `--wait-timeout` | number  | `30000`                            | Milliseconds to wait for navigation and selectors                                                                                    |
| `--source-maps`  | enum    | `auto`                             | `auto`, `on`, or `off`. Controls source map resolution for findings. Not yet implemented; all findings currently set `source: null`. |
| `--quiet`        | boolean | `false`                            | Suppress progress output to stderr                                                                                                   |

Mutual exclusion rules:

- `--url`, `--urls`, and `--build` without `--urls` are mutually exclusive
- `--build --urls` is the SPA build mode, both are required together
- `--threshold` only affects exit code, not report content

### 5.3 `foxhole compare`

```
foxhole compare <before.json> <after.json>
```

Flags:

| Flag          | Type   | Default    | Description                                       |
| ------------- | ------ | ---------- | ------------------------------------------------- |
| `--output`    | enum   | `markdown` | `json` or `markdown`                              |
| `--out`       | string | stdout     | File path for the diff report                     |
| `--threshold` | number | none       | Fail with exit code 1 if `score_delta` is below N |

Produces a `RunDiff` (see schemas spec) showing regressions, improvements, and unchanged findings.

### 5.4 `foxhole report`

```
foxhole report <report.json>
```

Re-renders an existing JSON report as markdown. Useful for CI workflows that store JSON and render later, or for re-rendering with future template changes.

Flags:

| Flag    | Type   | Default | Description                |
| ------- | ------ | ------- | -------------------------- |
| `--out` | string | stdout  | File path for the markdown |

### 5.5 `foxhole init`

Generates a `foxhole.config.json` in the current directory. Interactive prompts for the most common fields. Refuses to overwrite an existing config without `--force`.

Flags:

| Flag      | Type    | Default | Description                  |
| --------- | ------- | ------- | ---------------------------- |
| `--force` | boolean | `false` | Overwrite an existing config |

### 5.6 Examples

```
foxhole run --url https://example.com
foxhole run --urls https://example.com/login,https://example.com/dashboard
foxhole run --build ./dist --urls /login,/dashboard
foxhole run --url https://example.com --checks a11y,semantic
foxhole run --url https://example.com --output json --out ./audit.json
foxhole run --config foxhole.config.json --threshold 80
foxhole run --build ./dist --urls /,/products --concurrency 2 --perf-runs 3
foxhole compare ./before.json ./after.json
foxhole report ./audit.json --out ./audit.md
foxhole init
```

## 6. Configuration file

### 6.1 Shape

```json
{
  "url": "https://example.com",
  "urls": ["/login", "/dashboard", "/settings"],
  "build": "./dist",
  "checks": ["perf", "a11y", "semantic"],
  "output": "json",
  "out": "./audits/latest.json",
  "threshold": 80,
  "concurrency": 1,
  "perfRuns": 1,
  "perfProfile": "standard",
  "waitFor": null,
  "waitTimeout": 30000,
  "sourceMaps": "auto",
  "quiet": false
}
```

All fields are optional. Validation is performed by Zod. Unknown fields cause a hard error to prevent silent typos.

### 6.2 Resolution order

Settings are resolved in this order, last wins:

1. Built-in defaults
2. `foxhole.config.json` in the current working directory, if present
3. File specified by `--config`
4. CLI flags

This means CLI flags always override config files, and config files always override defaults.

### 6.3 Path resolution

Relative paths in the config file are resolved relative to the config file's location, not the current working directory. This allows configs to be referenced from any subdirectory.

## 7. Input modes

### 7.1 `--url` (single page)

Audits one URL. Simplest mode.

### 7.2 `--urls` (route list)

Comma-separated URLs or paths. If paths are given without `--build`, they are treated as relative to the host of the first absolute URL in the list. If no absolute URL is present, this is a hard error.

### 7.3 `--build` (local static build)

Foxhole spins up an internal HTTP server bound to an OS-assigned ephemeral port, serving the build directory via `serve-handler`. Used together with `--urls` to specify routes. The server is torn down when the audit completes.

Limitations:

- The internal server is static-file only. SPAs requiring server-side rendering or API calls will not work in `--build` mode without a separate backend running.
- No SPA fallback is applied. Routes must map to files that exist in the build directory; paths that do not resolve to a file receive a 404. Applications requiring SPA routing need to be audited against a running development server instead.

## 8. Checks

Four check categories. Each is independently enableable via `--checks`.

### 8.1 `perf`

Wraps Lighthouse. Reports Core Web Vitals (LCP, FID, CLS, FCP, TTFB, TBT), Lighthouse performance score, and key opportunities (oversized images, render-blocking resources, unused CSS, etc.).

### 8.2 `a11y`

Wraps axe-core. Reports accessibility violations with WCAG mappings, severity, and affected elements.

### 8.3 `semantic`

Custom DOM walker. Reports issues that axe does not catch:

- Heading order violations (skipped levels)
- Multiple h1 elements
- Landmark misuse (multiple main, nested landmarks)
- Generic link or button text ("click here", "read more")
- Inputs without associated labels (beyond what axe reports)
- Decorative images with non-empty alt
- Buttons styled as links and vice versa

### 8.4 `bundle`

Analyzes network requests captured during page load. Reports:

- Total transferred bytes (JS, CSS, images, fonts, other)
- Oversized JavaScript: total JS transfer size above 500KB, or any single JS file above 200KB (thresholds are hardcoded in this release)
- Oversized CSS: total CSS transfer size above 100KB (threshold is hardcoded in this release)
- Resources loaded over HTTP on an HTTPS page

Bundle analysis uses transfer size (compressed bytes over the wire), not decoded size. This choice is documented in findings output.

## 9. Output

### 9.1 Formats

Two formats. JSON is the canonical machine-readable form. Markdown is rendered from JSON.

The MCP server always returns JSON. The CLI defaults to markdown for human consumption.

### 9.2 JSON

Conforms to the `AuditReport` interface defined in the schemas spec. Versioned via the top-level `version: 1` field. Breaking changes increment this number.

### 9.3 Markdown

Rendered from the JSON via deterministic templates. Shape defined in the report template spec. Key properties:

- Top-of-report summary is mechanically generated from score, top finding, and regression count
- Top fixes table deduplicates findings across pages
- Per-page sections follow a fixed structure
- Cross-page duplicate findings link to their first occurrence
- Run metadata footer is always present

No prose generation, no LLM, no variability between runs on the same input.

### 9.4 Voice and style rules

These apply to all rendered output, all error messages, and all log lines:

- Plain ASCII punctuation only. No em dashes, no smart quotes.
- No AI tells. No "Certainly", "Great question", "I hope this helps", "Let's dive in", "In conclusion", "It's worth noting that".
- Senior engineer tone. Direct, factual, prioritized.
- No marketing language, no exaggerated claims, no emojis in output.

These rules are enforced by:

1. The findings catalog. Every recommendation string is hand-authored and reviewed.
2. The renderer. Templates contain no prose generation, only field interpolation.
3. Lint rules in CI. A custom check scans report templates and findings catalog entries for forbidden tokens.

## 10. Source map resolution

### 10.1 Behavior

> **Implementation status:** Source map resolution is not yet implemented. Every runner sets `source: null` unconditionally. The pipeline described below is the intended design for a later phase.

When `--source-maps` is `auto` or `on`, Foxhole attempts to resolve every finding to a source file location. The resolution chain:

1. Identify the runtime artifact responsible for the finding (DOM element, network request, etc.)
2. Locate the source map for the bundle containing that artifact
3. Resolve the runtime location through the source map to a source file path, line, and column
4. Read the source file (in `--build` mode, from disk; in `--url` mode, from the served path) to extract a snippet

If any step fails, the finding's `source` field is set to `null` and the finding still ships. Source map resolution failure is logged at debug level, never escalated to an error.

### 10.2 Modes

- `auto` (default): attempt resolution, silently fall back to null on failure
- `on`: attempt resolution, log a warning when it fails
- `off`: skip resolution entirely, faster but less actionable

### 10.3 Limitations

Documented in the README and surfaced in the report when relevant:

- Production URLs without source maps return null for `source` on most findings
- Source maps must be either inline, served alongside the bundle (`bundle.js.map`), or referenced via `sourceMappingURL` comment
- Cross-origin source maps are not fetched in v1
- The reported source location reflects the last build, not the current state of the file on disk. Stale source maps point at outdated lines.

### 10.4 Verification

Foxhole never reports a source location it cannot verify exists. In `--build` mode, the resolved file path must exist on disk relative to the build directory's project root, otherwise the location is discarded. This is the rule that prevents false positives from misconfigured builds.

## 11. Exit codes

| Code | Meaning                                                                                      |
| ---- | -------------------------------------------------------------------------------------------- |
| 0    | Audit completed, score met threshold (or no threshold set)                                   |
| 1    | Audit completed, score below threshold                                                       |
| 2    | Runtime error (browser launch failure, invalid config, network error preventing audit, etc.) |

Exit code 1 is reserved for "the audit ran, the result is bad". Exit code 2 is reserved for "the audit could not run". This distinction matters in CI.

## 12. MCP server

### 12.1 Transport

stdio. The MCP server is launched via `foxhole mcp` (subcommand to be confirmed in architecture spec) and communicates over stdin/stdout using the MCP protocol.

### 12.2 Tools

Six tools, all sharing the orchestrator with the CLI:

| Tool                      | Purpose                                                           |
| ------------------------- | ----------------------------------------------------------------- |
| `run_full_audit`          | Run all checks against a URL or URLs, return full `AuditReport`   |
| `run_accessibility_audit` | Run only the a11y check, return findings filtered to a11y         |
| `run_performance_audit`   | Run only the perf check, return findings and metrics              |
| `get_prioritized_fixes`   | Given an AuditReport JSON, return only the prioritized fixes list |
| `compare_runs`            | Compare two reports, return a `RunDiff`                           |
| `generate_report`         | Render a JSON report as markdown                                  |

### 12.3 Statelessness

The MCP server holds no persistent state. Each tool invocation is self-contained. Reports are not cached across calls in v1. If a tool needs a previous report (compare, get_prioritized_fixes), the report JSON is passed in the tool input.

This keeps the server simple and aligns with the "no server-side state without paying users" principle.

## 13. Performance budgets

These are commitments, not aspirations. Phase reviews check actual performance against these numbers.

| Scenario                                          | Target           |
| ------------------------------------------------- | ---------------- |
| Single URL, all checks, default profile           | Under 15 seconds |
| Single URL, a11y + semantic only                  | Under 5 seconds  |
| Single URL, perf only                             | Under 12 seconds |
| 10-route SPA via `--urls`, all checks, sequential | Under 90 seconds |
| MCP `run_accessibility_audit` call                | Under 5 seconds  |
| MCP `run_full_audit` for single URL               | Under 15 seconds |

Hardware reference: M-series Mac, 16GB RAM, gigabit connection to localhost or fast internet to public URLs. Slower hardware proportionally affects all numbers.

If a phase implementation blows a budget, that is a trigger to revisit, not a thing to ignore.

## 14. Performance and accuracy tradeoffs

The defaults optimize for the common case: a developer running locally or in CI on a single URL or small route list, wanting fast feedback, accepting that absolute perf scores are noisy.

Tunable knobs and their tradeoffs:

| Knob             | Default  | Faster                    | More accurate                    |
| ---------------- | -------- | ------------------------- | -------------------------------- |
| `--concurrency`  | 1        | Higher (page parallelism) | 1 (no resource contention)       |
| `--perf-runs`    | 1        | 1                         | 3 or 5 (median reduces variance) |
| `--perf-profile` | standard | fast (no throttling)      | mobile (full throttling)         |
| `--source-maps`  | auto     | off                       | on (fail loud)                   |
| `--checks`       | all      | subset                    | all                              |

Document these in the README and the spec. Do not hide them. Power users tune from defaults.

Known accuracy considerations:

- Lighthouse perf scores have inherent run-to-run variance of 3-5 points. Threshold-based CI failures should use a tolerance band, not exact matching.
- axe-core ruleset versions affect findings. The active version is recorded in `RunMeta.foxhole_version`'s dependency chain and surfaced in the report metadata.
- `networkidle + 500ms` is wrong for sites with persistent connections. `--wait-for` is the documented escape hatch.
- Findings are identified by stable hashes (rule id + nearest semantic ancestor + text content), not CSS path. This makes `compare_runs` resilient to layout changes.

## 15. Acceptance criteria for v1.0.0

These are the conditions under which v1.0.0 ships.

### Functionality

- All four commands implemented and documented
- All input modes (`--url`, `--urls`, `--build`, `--build --urls`) functional
- All four checks (`perf`, `a11y`, `semantic`, `bundle`) functional
- Both output formats (`json`, `markdown`) produce conforming output
- All six MCP tools implemented and tested
- Source map resolution working in `--build` mode end-to-end (not yet implemented)
- Source map resolution working for public URLs that serve maps (not yet implemented)
- Exit code contract honored
- Config file resolution working with documented precedence

### Quality

- Performance budgets met on reference hardware
- Test coverage above 80% for the audit, score, prioritize, diff, and normalization modules
- Integration tests covering each input mode, each output format, and each MCP tool
- Zero `any` types in the public API surface
- All four logged technical debt items resolved (build flag wired, Result type consistent, single Chrome instance, axe effort estimation)

### Documentation

- README with installation, quickstart, and all commands
- This spec, the architecture spec, the schemas spec, and the report template spec all current
- Findings catalog populated for all rules that ship
- ADRs for any architectural decisions made during implementation
- CHANGELOG.md with v1.0.0 entry

### Distribution

- Published to npm as `foxhole`
- Binary available globally via `npm install -g foxhole`
- Node 20 LTS minimum, tested on 20 and 22
- macOS, Linux, and Windows tested in CI

## 17. Open questions

These are questions whose answers go in the architecture spec, the schemas spec, or a phase spec. They are listed here so they are not forgotten.

1. MCP server invocation: standalone subcommand `foxhole mcp` or separate binary? Architecture spec decides.
2. Internal static server: Node `http` module or `serve-handler`? Architecture spec decides.
3. Source map resolution library: `source-map` (Mozilla) or `@jridgewell/trace-mapping` (faster)? Architecture spec decides.
4. Lighthouse integration: programmatic via `lighthouse` package, or `lighthouse-ci` runner? Architecture spec decides.
5. Findings catalog format: TypeScript file, JSON, or YAML? Schemas spec or a dedicated catalog spec decides.
6. How to express "this finding affects N pages" in the JSON without bloating per-page arrays. Schemas spec decides.
